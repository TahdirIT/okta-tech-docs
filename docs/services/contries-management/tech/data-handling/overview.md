# التعامل التقني مع البيانات (Data handling) — نظرة عامة

هذا القسم يوضح **النهج التقني** لتخزين/قراءة/التحقق من بيانات خدمة **إدارة الدول**، خصوصًا البيانات التي تكون:

- مرنة ومتغيرة (Settings)
- تعتمد على تعريفات ديناميكية (مثل: حقول مخصصة)
- تتطلب تمثيلًا مناسبًا لتعدد اللغات (i18n)

## 1) متى نستخدم JSON (jsonb)؟

نستخدم `jsonb` عندما تكون البنية:

- متغيرة حسب الدولة (Country-level variability)
- قابلة للتوسع بدون تغييرات Migration متكررة
- تحتوي على مصفوفات/كائنات متداخلة (Nested objects/arrays)

في خدمة إدارة الدول، يُقترح حفظ أغلب إعدادات الدولة داخل:

- `model_settings.settings` (PostgreSQL `jsonb`)

## 2) ما الذي يجب أن يكون Columns / جداول؟

يفضل جعلها أعمدة/جداول عندما:

- نحتاج استعلامات DB متكررة/ثقيلة على قيم محددة
- نحتاج قيود DB قوية (FK/unique/check) بدل الاعتماد على التطبيق فقط
- نحتاج تقارير/فلترة/فرز على نطاق واسع وبأداء عالٍ

> يمكن اتباع حل وسط: JSON كمصدر أساسي + أعمدة مشتقة (Derived columns) أو Views عند ظهور احتياج فعلي.

## 3) قواعد عامة للتعامل مع إعدادات الدولة داخل `model_settings`

- **Defaulting**: أي مفاتيح غير موجودة تُفسّر بقيم افتراضية واضحة (Documented defaults)
- **Validation**: التحقق يكون على مستوى التطبيق (Form Request / Service) مع رسائل أخطاء مناسبة
- **Backward compatibility**: عند تغيير الـSchema، نراعي بقاء القراءة متوافقة مع البيانات القديمة قدر الإمكان
- **Indexing**: إذا احتجنا استعلامات على `settings->...` نضيف فهارس مناسبة (GIN/Expression indexes) حسب الاستخدام الفعلي
