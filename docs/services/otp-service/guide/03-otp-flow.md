# تدفق OTP (Request / Verify / Consume)

هذا المستند يوضح التدفق المتوقع على مستوى الاستخدام (Use-case) دون الدخول في تفاصيل تقنية التنفيذ.

## 1) Request OTP

- المُستهلك (Client/Service) يطلب إنشاء OTP لغرض محدد `purpose` ووجهة `destination` (مثل رقم جوال أو بريد).
- الخدمة تتحقق من السياسات:
  - حدود الطلبات (Rate limiting).
  - صلاحية القناة والغرض.
- يتم إنشاء رمز مع **TTL** وتسجيله وربطه بالسياق.
- يتم إرسال الرمز عبر قناة الإشعار المناسبة.

## 2) Verify OTP

- المُستهلك يرسل `otp_code` للتحقق مع نفس `purpose` و `destination`.
- الخدمة تتحقق من:
  - تطابق الرمز وعدم انتهائه.
  - عدم تجاوزه لمحاولات التحقق.
  - عدم كونه مستهلكاً مسبقاً.
- النتيجة تكون **نجاح/فشل** مع سبب واضح للفشل (expired/invalid/locked/rate_limited).

## 3) Consume OTP (اختياري)

حسب سياسة النظام يمكن فصل التحقق عن الاستهلاك:

- **Verify**: إثبات صحة الرمز.
- **Consume**: تعليم الرمز كمستخدم حتى لا يعاد استخدامه.

هذا يفيد في تدفقات تتطلب خطوة لاحقة بعد التحقق (مثل إنشاء جلسة/تغيير كلمة مرور).
