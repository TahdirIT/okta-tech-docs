# الحالات الحدّية (Edge Cases) — Tenant Members Management

## 1) طالب موجود مسبقاً

- **الحالة**: إدخال هوية طالب موجود ولديه سجل طالب فعّال داخل نفس الجهة.
- **المطلوب**: منع الإضافة وإظهار رسالة “الطالب مرتبط بالفعل بهذه الجهة”.

## 2) طالب مرتبط بجهة أخرى

- **الحالة**: إدخال هوية طالب فعّال في جهة مختلفة وفق سياسة النظام.
- **المطلوب**:
  - منع الإضافة.
  - توضيح أن الطالب مرتبط بجهة أخرى (بدون تسريب بيانات الجهة الأخرى).

## 3) طالب بدون فصل فعّال

- **الحالة**: محاولة حفظ طالب دون تحديد فصل.
- **المطلوب**: منع الحفظ بشكل صارم (UI + API).

## 4) نقل طالب ثم فشل حفظ جزئي

- **الحالة**: تحديث فصل الطالب، وتم release للرابط القديم لكن attach الجديد فشل.
- **المطلوب**:
  - تنفيذ العملية داخل transaction.
  - أو ضمان تعويض (compensation) يعيد الطالب لحالة متسقة.

## 5) فصل طالب من الجهة مع بقاء روابط أخرى

- **الحالة**: فصل الطالب من الجهة مع وجود روابط ولي/سجلات أخرى.
- **المطلوب**:
  - فصل العضوية الأكاديمية (Section) أولاً أو معها.
  - ترك روابط ولي الأمر كسجل تاريخي (أو وفق سياسة المنتج).

## 6) فصل ولي عن طالب ثم إعادة الربط

- **الحالة**: كان هناك `unlinked_at` ثم إعادة الربط.
- **المطلوب**:
  - إما إنشاء سجل علاقة جديد
  - أو إعادة تنشيط السجل القديم (إفراغ unlinked_at) حسب سياسة التدقيق.

## 7) استيراد ملف بصف/فصل غير موجودين

- **الحالة**: stage/section في الملف غير معرفين للجهة.
- **المطلوب**:
  - إما منع التنفيذ وإظهار أخطاء مع إرشاد لإنشاء الصفوف/الفصول
  - أو السماح بإنشائها تلقائياً ضمن سياسة منتج واضحة

## 8) تكرار روابط Pivot

- **الحالة**: وجود أكثر من سجل pivot فعّال (released_at = null) لنفس الطالب↔جهة أو الطالب↔فصل.
- **المطلوب**:
  - Jobs تنظيف (dedup) بعد الاستيراد
  - ومنع ذلك عبر قيود قاعدة البيانات إن أمكن
